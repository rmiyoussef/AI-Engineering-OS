# Frontend Design Principles

> **Source:** Adapted from anthropics/skills (frontend-design)
> **Domain:** Frontend
> **Use when:** Creating distinctive, non-templated visual identities for user-facing projects.

---

## Overview

Design principles for creating visual identities that stand out from AI-generated templates. The goal is work that looks intentionally designed, not procedurally generated.

## Grounding

Before designing, name three things:
1. **The subject** — what exactly are we designing?
2. **The audience** — who is this for?
3. **The purpose** — what should this page/screen accomplish?

## Design Principles

### The Hero is the Thesis
The first thing a user sees should immediately communicate what the page is about. It's not decoration — it's the thesis statement.

### Typography Carries Personality
Type choices (family, weight, size, leading, measure) do more heavy lifting for brand personality than color ever will. Nail the type hierarchy before adding visual flourishes.

### Structure Information Purposefully
Don't let information design be an afterthought. Group related content, create clear visual hierarchies, and guide the eye from most to least important.

### Use Motion Deliberately
Motion should serve a purpose: guide attention, show state change, or add delight. Never animate just because you can.

### Match Complexity to the Vision
A simple landing page doesn't need a complex design system. A data dashboard doesn't need heavy illustration. Match the level of design polish to the actual need.

## Process

### 1. Brainstorm a Token System
Define constraints before creating:
- **Color** — palette limited to 3-5 colors
- **Type** — 2-3 families, clear hierarchy
- **Layout** — grid, spacing scale, breakpoints
- **Signature** — one distinctive element (illustration style, border treatment, motion signature)

### 2. Review Against the Brief
Every design decision should trace back to a requirement. If it doesn't, question whether it needs to be there.

### 3. Build
Create within the token system. Go back to the brief frequently.

## Restraint & Self-Critique

- Spend boldness in one place, not everywhere
- Be responsive and accessible from the start (not retrofitted)
- Ask: "Would I notice if this element was removed?"

## Writing Guidance

- Active voice, conversational register
- Name things from the user's perspective (not the system's)
- Short labels, clear calls-to-action
